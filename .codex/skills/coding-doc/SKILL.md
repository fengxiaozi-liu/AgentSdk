# Coding Doc

使用本技能，将代码、架构讨论或改造方案转换成适合 AI 编程代理理解和继续开发的设计文档。

目标是在保持 Markdown 清晰、美观、便于人类审阅的同时，让另一位 AI 编程代理可以稳定翻译成 Go、Java、Python、TypeScript、SQL、测试或重构方案，并且不会误解模块边界。

## 核心流程

1. 先判断文档类型，再选择结构。
2. 只读取理解目标模块所需的代码和文档。
3. 提炼组件、实体、流程、契约、决策、不变量和扩展点。
4. 根据模块类型选择图表表达方式。
5. 使用编号章节、表格、Mermaid 图和明确约束编写文档。
6. 除非用户明确要求计划，否则不要写实施步骤。

## 文档类型选择

| 用户意图 | 文档类型 | 是否包含当前问题 | 是否包含目标架构 |
| --- | --- | --- | --- |
| 理解现有代码仓库 | 仓库架构文档 | 默认不包含 | 默认不包含 |
| 分析问题或规划重构 | 改造设计文档 | 包含 | 包含 |
| 设计新功能 | 功能设计文档 | 可选 | 包含 |
| 记录设计决策 | 决策记录文档 | 可选 | 可选 |
| 为后续 AI 开发准备上下文 | AI 编程上下文文档 | 按需包含 | 按需包含 |

当用户只是要求理解现有代码仓库时，不要强行加入“当前问题”或“目标架构”章节。

## CodingDoc 语法

`references/coding-doc-syntax.md` 是规范语法。创建或大幅修改设计文档时，应加载并遵循它。

核心块如下：

| 块 | 作用 |
| --- | --- |
| 上下文 | 定义范围、目标、非目标、读者和假设。 |
| 组件 | 描述模块职责、依赖、归属和非职责。 |
| 实体 | 描述数据结构、字段、语义和约束。 |
| 关系 | 展示实体如何连接，通常使用 E-R 图。 |
| 流程 | 描述运行时行为、分支和输出。 |
| 契约 | 定义 API、方法、事件、查询或协议行为。 |
| 决策 | 记录采用和放弃的设计选择。 |
| 不变量 | 声明实现必须保持的规则。 |
| 扩展点 | 标记未来扩展方向，不把它变成当前需求。 |

## 表达模式选择

根据模块关注点选择表达方式：

| 模块关注点 | 推荐表达 |
| --- | --- |
| 模块边界与依赖 | 模块架构图 + 职责表 |
| 数据模型 | E-R 图 + 实体字段表 |
| 请求执行路径 | Run 流程图 + 步骤表 |
| 查询/过滤行为 | 查询流程图 + 条件表 |
| 生命周期/状态逻辑 | 状态流转图 + 状态表 |
| 设计取舍 | 决策表 |
| 公共 API 或服务方法 | 契约表 |
| 必须保持的规则 | 不变量表 |

选择 Mermaid 图类型时，参考 `references/diagram-patterns.md`。

## 模板

从 `assets/templates/` 中选择最接近用户需求的模板作为起点，然后按上下文调整。

| 模板 | 适用场景 |
| --- | --- |
| `repo-architecture.md` | 解释现有代码仓库。 |
| `refactor-design.md` | 分析问题并规划目标架构。 |
| `feature-design.md` | 设计新能力。 |
| `module-design.md` | 记录单个 package/module/service。 |
| `data-model-design.md` | 聚焦实体、存储和查询语义。 |

## 编写规则

- 优先使用表格和 Mermaid 图，少写大段散文。
- 使用编号章节：`1.`、`1.1`、`1.2`。
- 只有在用户要求问题分析或重构时，才包含当前问题。
- 只有在用户明确要求计划时，才包含实施步骤。
- 未确认想法放入假设、待确认问题或扩展点，不要写成最终决策。
- 每个核心实体都应包含字段、含义、约束和关系。
- 每个流程都应包含入口、分支和终止输出。
- 每个查询都应定义默认行为和特殊条件。
- 每个决策都应包含采用方案、放弃方案、原因和影响。
- 每个不变量都应写成实现必须保持的规则。

## 质量标准

一份合格的 CodingDoc 应让后续 AI 编程代理回答这些问题：

| 问题 | 文档中应提供的答案 |
| --- | --- |
| 这个行为由哪个模块拥有？ | Component 和边界章节。 |
| 需要哪些数据结构？ | Entity 和 Relation 章节。 |
| 运行时发生什么？ | Flow 章节。 |
| 每个 API/查询默认做什么？ | Contract 章节。 |
| 哪些选择已经确定？ | 决策章节。 |
| 哪些规则不能被破坏？ | 不变量章节。 |
| 哪些内容被有意推迟？ | 非目标和扩展点章节。 |
